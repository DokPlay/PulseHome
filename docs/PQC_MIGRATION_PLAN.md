# План миграции на постквантовую криптографию (PQC) для проекта PulseHome

Этот документ описывает пошаговую стратегию внедрения квантово-устойчивых алгоритмов в платформу `PulseHome`. План основан на анализе архитектуры проекта и лучших практиках, изложенных в предоставленных документах по PQC.

## 🎯 Основные цели

1.  **Защита от атак "Store Now, Decrypt Later" (SNDL)**: Защитить данные, передаваемые от внешних устройств, от перехвата и последующей расшифровки на квантовом компьютере.
2.  **Долгосрочная аутентичность**: Гарантировать, что команды и обновления прошивки не могут быть подделаны в будущем с помощью квантовых атак.
3.  **Плавная миграция**: Обеспечить обратную совместимость и минимизировать риски, связанные с внедрением новых, менее изученных алгоритмов.

## 🔑 Ключевая стратегия: Гибридный подход

Мы не будем заменять классическую криптографию (ECC, RSA), а **дополним** её постквантовой. Это стандарт де-факто для миграции.

- **Обмен ключами**: `X25519` (классический) + `ML-KEM-768` (постквантовый). Злоумышленнику для взлома сессии потребуется взломать **оба** алгоритма.
- **Цифровая подпись**: `ECDSA` (классический) + `ML-DSA-65` (постквантовый).

Этот подход решает сразу две проблемы: сохраняет производительность и совместимость, а также страхует от потенциальных уязвимостей в новых PQC-алгоритмах.

## 🗺️ Пошаговый план внедрения

### Шаг 0: Подготовка фундамента ("Крипто-гибкость")

**Цель**: Подготовить проект к изменениям и избежать жёсткой привязки к конкретным алгоритмам.

1.  **Аудит криптографии (CBOM)**:
    - Проанализировать все точки использования криптографии:
        - TLS-терминация в `Collector` (встроенный веб-сервер).
        - gRPC-клиент в `Analyzer` для связи с `Hub Router`.
        - Kafka-клиенты во всех сервисах (`Collector`, `Aggregator`, `Analyzer`).

2.  **Обновление библиотек**:
    - Добавить в корневой `pom.xml` проекта зависимости **Bouncy Castle**. Эта библиотека является промышленным стандартом, предоставляет FIPS-сертифицированные реализации и поддерживает гибридные схемы, что идеально подходит для нашего плана.
      ```xml
      <dependency>
          <groupId>org.bouncycastle</groupId>
          <artifactId>bcprov-jdk18on</artifactId>
          <version>1.79</version> <!-- или новее -->
      </dependency>
      <dependency>
          <groupId>org.bouncycastle</groupId>
          <artifactId>bcpqc-jdk18on</artifactId>
          <version>1.79</version> <!-- или новее -->
      </dependency>
      ```

3.  **Регистрация провайдера**:
    - Во всех сервисах (`Collector`, `Aggregator`, `Analyzer`) при старте приложения необходимо статически зарегистрировать провайдер Bouncy Castle:
      ```java
      import java.security.Security;
      import org.bouncycastle.jce.provider.BouncyCastleProvider;
      import org.bouncycastle.pqc.jcajce.provider.BouncyCastlePQCProvider;

      // В главном классе приложения или в @PostConstruct
      Security.addProvider(new BouncyCastleProvider());
      Security.addProvider(new BouncyCastlePQCProvider());
      ```

### Шаг 1: Защита внешних каналов (Высший приоритет)

**Цель**: Защитить самые уязвимые точки — входящие HTTP-запросы и исходящие gRPC-команды.

1.  **`Collector` (HTTPS Ingestion)**:
    - **Задача**: Защитить TLS-рукопожатие при подключении хабов и датчиков.
    - **Решение**: Настроить встроенный в Spring Boot веб-сервер (Tomcat/Netty) на использование гибридной схемы обмена ключами в TLS 1.3.
    - **Алгоритм**: `X25519MLKEM768` (стандартное название для гибридной схемы `X25519` и `ML-KEM-768`).
    - **Реализация**: Это делается через конфигурацию `SSLParameters` или аналогичные настройки JSSE. Потребуется настроить Java для использования JSSE-провайдера от Bouncy Castle, который поддерживает эту гибридную группу.

2.  **`Analyzer` → `Hub Router` (gRPC)**:
    - **Задача**: Защитить команды, отправляемые на `Hub Router`.
    - **Решение**: Аналогично настроить gRPC-клиент в `Analyzer` и gRPC-сервер в `Hub Router` на использование гибридного TLS 1.3.
    - **Реализация**: gRPC позволяет настраивать `SslContext`. Нужно будет сконфигурировать его для использования гибридной схемы `X25519MLKEM768`.

### Шаг 2: Усиление аутентификации и целостности команд

**Цель**: Заменить слабую `Basic Auth` на криптографическую подпись, устойчивую к квантовым атакам.

**Задача**: Гарантировать, что каждая команда, приходящая в `Collector`, отправлена доверенным устройством и не была изменена.

**Решение**: Внедрить схему **"Подпиши, затем зашифруй" (Sign-then-Encrypt)** на уровне приложения. Это более гибко, чем mTLS, и идеально соответствует примерам из ваших документов.

1.  **На стороне устройства (клиента)**:
    - Сгенерировать долгосрочную пару ключей для подписи: `ML-DSA-65`.
    - Перед отправкой данных в `Collector`:
        1.  Подписать `(payload)` с помощью приватного ключа `ML-DSA`.
        2.  Создать сообщение `(payload + signature)`.
        3.  Выполнить обмен ключами `ML-KEM-768` с публичным ключом `Collector`.
        4.  Зашифровать сообщение `(payload + signature)` с помощью полученного сессионного ключа (например, через AES-256-GCM).
        5.  Отправить `(encapsulation + encrypted_message)` на `Collector`.

2.  **На стороне `Collector`**:
    - Получив запрос:
        1.  Используя свой приватный ключ `ML-KEM`, извлечь сессионный ключ из `encapsulation`.
        2.  Расшифровать `encrypted_message` и получить `(payload + signature)`.
        3.  Проверить подпись `signature` с помощью публичного ключа `ML-DSA` устройства.
        4.  Если подпись верна, обработать `payload`.

**Преимущество**: Этот подход обеспечивает и конфиденциальность (шифрование), и аутентичность/целостность (подпись) на постквантовом уровне.

### Шаг 3: Защита внутреннего трафика (Kafka)

**Цель**: Защитить данные, передаваемые между сервисами внутри платформы.

**Задача**: Зашифровать трафик между `Collector`, `Aggregator`, `Analyzer` и брокером Kafka.

**Решение**: Включить и настроить TLS для всех Kafka-клиентов.

**Реализация**: В `application.properties` каждого сервиса настроить SSL для Kafka, указав использование TLSv1.3 и, если возможно, настроив JSSE для поддержки гибридных схем, как в Шаге 1. Даже без гибридных схем, простое включение TLS уже является значительным улучшением безопасности.

### Шаг 4: Управление ключами и развертывание

**Цель**: Обеспечить управляемый и безопасный жизненный цикл ключей.

1.  **Разработка**: Начать с пилотной реализации в `dev`-окружении.
2.  **Управление ключами**: Разработать процедуры для генерации, ротации и отзыва PQC-ключей. На начальном этапе это могут быть скрипты, использующие Bouncy Castle. В будущем — интеграция с PKI-системами (Key Management System), поддерживающими PQC.
3.  **Мониторинг**: Внимательно следить за производительностью (размер ключей и подписей PQC больше, что может влиять на задержки в сети) и логами на предмет ошибок TLS-рукопожатия.
